## A. 设计语言与原则

M1 面向中文创作者,气质应**克制、内容优先、低装饰**(与"留白 / 间"的审美一致),让注意力落在**被审的内容**上,而不是工具本身。

**7 条 UX 原则:**
1. **管线即导航**:顶部一条固定的阶段进度条(created→…→exported)是全站主轴,永远显示"你在哪一步 + 唯一主 CTA"。
2. **每步只审一件事**:文案 / 语音 / 画面各一屏,**每屏只有 3 个主按钮:批准 · 重新生成 · 手动编辑**。不堆功能。
3. **停下等人审是一等状态**,不是错误:`awaiting_review` 用平静的中性色呈现,给"批准/改"的明确出口。
4. **失效要显性、可解释**:审批因内容改动失效时,用醒目但不惊吓的横幅说明"改了什么 → 哪些审核失效 → 下一步"。
5. **预览低清、真相在后端**:右侧预览是草稿质量,明确标"草稿";真值永远是后端 artifact。
6. **中文排版讲究**:UI 与字幕都用 CJK 字体;字幕遵循中文断行规则(见 §D)。
7. **危险操作留痕不隐藏**:release 前展示 provider/license/QA 阻塞状态;没有任何"绕过审核"的入口。

**视觉基调(建议):** 大量留白、单一强调色、无多余边框/阴影;暗/亮双色;正文中文字体优先(思源黑/系统 CJK);状态用色语义化(待审=中性、通过=正向、失效/阻塞=警示)。

├───────────┬───────────────────────────────────┬───────────┤

## C. 五页逐屏规范(目的 · 所见 · 动作 · 状态 · 组件)

### 页 1 · 新建向导
- 目的:最少摩擦开工。**默认只问两件事:视频类型 + 目标平台**;其余(语言/比例/时长/风格/受众/品牌资料)折叠为高级。
- 所见:输入区(文本/URL/截图/图片)+ doctor 摘要(FFmpeg/字体/LLM/TTS/OCR extras/Playwright/Remotion license 提示)。
- 动作:创建并进入提取。
- 状态:doctor 有 hard 缺失(如无 FFmpeg/无 CJK 字体)→ 阻塞并给一键修复命令;可选缺失(OCR/Playwright)→ 黄色提示不阻塞。

### 页 2 · 提取 + 文案审核
- 目的:确认"机器读懂了什么" + 审文案。
- 所见:原始输入、提取正文、metadata、截图预览;`script.json` 的分镜/字幕文本/**source_map(来源)**;标题/正文/话题。
- 动作:**批准 · 重新生成 · 手动编辑**;中英切换;版本对比。
- 状态:输入过薄(正文<阈值/OCR 空)→ 提示补充;编辑后重算 script hash,**旧审批即时失效并提示**。

### 页 3 · 语音审核
- 目的:听着满意再往下。
- 所见:provider/voice、每段音频、**逐段实测时长 + 总时长**、字幕同步检查。
- 动作:试听 · 换声音 · 重生成;批准。
- 状态:审批绑定 `voice_plan.json` + 音频文件清单 hash;重跑 `lj voice` = 新音频 → 需重新审批(显性提示);TTS 超时/限流 → 明确错误码不伪装成功。

### 页 4 · 画面审核
- 目的:确认画面表达 + 排版。
- 所见:`visual_plan.json`、scene 卡片、比例预览、安全区、字体状态、截图高亮。
- 动作:低清预览(`lj preview`)· 编辑布局 · 重生成;批准。
- 状态:审批绑定 visual plan hash;字体缺失 → 阻塞并提示下载。

### 页 5 · 渲染 + 发布包
- 目的:出片、体检、下载。
- 所见:三审状态、render 按钮、QA 报告(hard/warn/info)、导出目录、provider/license manifest。
- 动作:渲染 · 查看 QA · 导出(单平台 / `--all-platforms`)· 下载。
- 状态:**任一审核未通过或失效 → render 被拒并指明缺哪项**;QA 有 hard fail 或含 mock → release 导出被拒并说明原因;doctor/provider 配置作设置抽屉,不单独拆页。

## D. 审核门禁 UX(产品命门,必须做对)

- **awaiting_review**:生成完成后,主区顶部平静提示"已生成草案,请审核",主 CTA=批准,次=重新生成/手动编辑。**系统不自动前进。**
- **approval stale(失效)**:当脚本/语音计划/画面计划被改,受影响审批失效。横幅文案示例:
  > 「脚本已修改。语音与画面的审核已失效,渲染前需重新审核。」并高亮失效的步骤 + 提供"去重审"跳转。
- **render blocked**:未审/失效点 render → 内联错误卡:"渲染被拦截:画面尚未审核 / 脚本审核已失效",给出到对应审核页的按钮。**无"强制渲染"入口。**
- **release blocked**:release 导出前,若 QA 有 hard fail 或 provider_manifest 含 mock → 阻塞卡说明:"当前使用了 mock provider,不能作为正式发布导出;请配置真实 LLM/TTS 后重试",或列出 QA hard 失败项。
- **CJK / 字体缺失**:阻塞卡 + 一键"下载中文字体"。

## E. 状态目录(每屏都要有的 5 态 + 降级)

| 状态 | 呈现 | 处理 |
|---|---|---|
| 空(empty) | 引导下一步(如"粘贴文本或上传截图开始") | 不显示假数据 |
| 加载(loading) | 步骤级骨架屏 + 当前动作说明(避免夸张动效,尤其严肃内容) | 不阻塞其它可读区 |
| 待审(awaiting_review) | 中性提示 + 3 主按钮 | 系统不自动前进 |
| 失效(stale) | 醒目横幅 + 失效步骤高亮 + 去重审 | 见 §D |
| 错误(error) | 内联可读错误 + 错误码 + 修复建议 + 重试 | 不吞错、不伪成功 |
| 降级(degraded) | 明示"部分能力不可用"(如无 OCR extras) | 给出安装指引,主流程继续 |
| 成功(success) | 明确产物 + 下载/下一步 | 不长篇说教 |

## G. 业务流(泳道:创作者 / 团队 / Agent × 决策门 × 失败分支)

```text
输入 → 提取 → 脚本 →[文案审]→ 语音 →[语音审]→ 画面 →[画面审]→ 渲染 → QA → 导出
                     │              │              │            │
   失败/失效分支： 改文案→失效     换声音→重审     改布局→重审   QA hard→回退/修  release 含 mock→拒绝
```

- **创作者泳道(个人)**:一人走完三审 → 渲染 → 导出。系统在每个 [审] 处停。
- **团队泳道(协作)**:运营建项目 + 输入 → PM 审文案 → 主理人审语音 → 品牌审画面 → 运营渲染/多平台导出。**每次审批记录 who + version(artifact hash)+ when**,写入 `export_manifest`,可审计。任一上游改动 → 下游失效,泳道回流。
- **Agent 泳道(Codex/Claude Code)**:Agent 调 `lj` 生成类命令 → 命中 `awaiting_review` **必须停** → 把审核链接/artifact 路径交给人 → 人审(Web 或 `lj approve`)→ Agent 续渲染/导出。Agent **不能自动 approve、不能用 preview 冒充 release、不能绕门禁**。
